# System Architecture

## High-level layout

```text
Browser (Admin/Faculty/Student)
  -> Next.js App Router (src/app)
  -> API Routes (src/app/api)
  -> MongoDB (via Mongoose models)
  -> Python AI Service (ai-service, FastAPI)
```

## Frontend and API

- App pages are in src/app with role-based areas:
  - /admin/*
  - /faculty/*
  - /student/*
- API routes are in src/app/api and include:
  - auth
  - classes
  - attendance
  - students
  - faculty
  - classrooms
  - monitor proxies

## Data and tenancy

- Mongoose models are in src/models.
- Each request resolves tenant context and applies institution scope.
- Class, student, faculty, and attendance data are institution-aware.

## Monitoring flow

1. Admin chooses class and source in /admin/camera.
2. Frontend calls same-origin monitor proxy routes:
   - POST /api/monitor/start
   - POST /api/monitor/stop
   - GET /api/monitor/status/:classId
   - GET /api/monitor/active
3. Proxy routes call AI service at AI_SERVICE_URL.
4. Detection results can be posted to:
   - POST /api/attendance/mark

## Class status synchronization

- Status values are synchronized based on class time windows.
- Synchronization runs before class list responses in main class APIs.

## Deployment shape

- One Next.js server process
- One AI service process
- MongoDB Atlas cluster
- Optional RTSP camera infrastructure
